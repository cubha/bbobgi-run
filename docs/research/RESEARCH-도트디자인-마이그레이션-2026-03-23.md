## 1. 리서치 배경

뽑기런의 전체 UI/그래픽을 현재 벡터+그라디언트 스타일에서 **도트(픽셀아트)** 디자인으로 전환하기 위한 기술 조사.
현재 코드베이스는 PixiJS v8 Graphics API로 실시간 벡터 드로잉 중이며, 이를 레트로 8bit 풍으로 교체하는 최적 전략을 수립한다.

### 현재 상태 요약

| 항목 | 현재 설정 | 문제점 |
|---|---|---|
| `antialias` | `true` | 픽셀아트에 부적합 + 성능 저하 (v8 버그 #10413) |
| `resolution` | `devicePixelRatio` | 고해상도 렌더링 → 픽셀 경계 흐림 |
| 텍스처 필터링 | 기본값 (`linear`) | 스프라이트 업스케일 시 보간으로 흐릿 |
| 엔티티 렌더링 | `Graphics.clear()` + 매 프레임 재드로잉 | CPU 오버헤드 + 도트 스타일 불가 |
| 폰트 | Black Han Sans, Noto Sans KR | 벡터 폰트 → 도트 느낌 없음 |
| 색상 | 그라디언트+글로우 기반 | 레트로 팔레트와 괴리 |

## 2. 조사 결과

### 2-1. PixiJS v8 픽셀아트 렌더링 설정

#### 핵심 3가지 설정

```typescript
import { TextureStyle } from 'pixi.js';

// 1) 전역 nearest neighbor — 반드시 Assets.load() 이전에 호출
TextureStyle.defaultOptions.scaleMode = 'nearest';

// 2) Application 초기화
await pixi.init({
  background: COLORS.background,
  resizeTo: container,
  antialias: false,       // 픽셀아트 필수
  resolution: 1,          // 저해상도 고정
  autoDensity: true,
  roundPixels: true,      // 좌표를 Math.floor() → 서브픽셀 렌더링 방지
});

// 3) CSS 업스케일 선명도
pixi.canvas.style.imageRendering = 'pixelated';
```

#### scaleMode 비교

| 모드 | 동작 | 용도 |
|---|---|---|
| `'linear'` (기본) | 보간으로 부드럽게 확대 | 사진, HD 그래픽 |
| `'nearest'` | 가장 가까운 픽셀 복제 | **픽셀아트 필수** |

#### roundPixels의 효과

`roundPixels: true` 설정 시 모든 오브젝트의 렌더링 좌표가 정수로 내림 처리되어, 텍스트/스프라이트가 0.5px 어긋나서 흐릿하게 보이는 현상을 완전 제거한다.

#### CSS image-rendering 브라우저 호환성

| 브라우저 | 지원 값 |
|---|---|
| Chrome / Edge | `pixelated` |
| Firefox | `crisp-edges` |
| Safari | `pixelated` (최신) |

> **주의**: CSS `pixelated`는 **정수 배율** 업스케일에서 가장 선명. 소수점 배율(1.5x 등)에서는 픽셀 크기 불균일 발생.

#### PixiJS v8 `pixelLine` 속성

v8에서 추가된 Graphics 전용 속성으로, 줌/스케일과 무관하게 **항상 1스크린 픽셀** 두께를 유지한다. 도트 스타일 외곽선/그리드에 핵심.

```typescript
graphics.stroke({ color: 0xffffff, pixelLine: true });
```

### 2-2. 저해상도 캔버스 업스케일 전략

#### 전략 A: CSS 전체 캔버스 업스케일 (권장)

```typescript
await pixi.init({
  width: DESIGN_WIDTH,   // 390
  height: DESIGN_HEIGHT, // 844
  antialias: false,
  resolution: 1,
  autoDensity: true,
});

pixi.canvas.style.width = '100%';
pixi.canvas.style.height = '100%';
pixi.canvas.style.imageRendering = 'pixelated';
```

- 가장 단순, 현재 프로젝트의 `resizeTo` 패턴과 호환
- 논리 해상도(390x844)가 이미 모바일 기준이라 별도 축소 불필요

#### 전략 B: RenderTexture 중간 버퍼 (고급)

```typescript
const lowResRT = RenderTexture.create({
  width: 195, height: 422,  // 절반 해상도
  scaleMode: 'nearest',
});
// 매 프레임: 저해상도로 렌더 → 전체 화면 스프라이트로 업스케일
```

- "진짜 저해상도 도트" 느낌 극대화
- 구현 복잡도 높고 UI 인터랙션(터치 좌표 변환) 추가 작업 필요

**결론**: 전략 A 채택 권장. 현재 DESIGN_WIDTH(390)에서 충분한 도트감 확보 가능.

#### 전환 옵션 비교

| 방식 | 장점 | 단점 | 적합도 |
|---|---|---|---|
| **A) AnimatedSprite + 스프라이트시트** | 최고 성능, 도트 완성도 최고, GPU 캐시 | 에셋 제작 필요 (Aseprite) | **최적** |

1. **즉시**: Graphics rect 기반으로 간단한 도트 캐릭터 프로토타입
2. **단기**: GraphicsContext 프레임별 사전 빌드 + `cacheAsTexture()`
3. **중기**: Aseprite로 스프라이트시트 제작 → AnimatedSprite 교체

## 3. 프로젝트 적합성 분석

### 호환성

- PixiJS v8은 `TextureStyle.defaultOptions.scaleMode = 'nearest'`, `roundPixels`, `pixelLine` 등 **픽셀아트 네이티브 지원이 우수**
- 현재 코드의 `Graphics` API는 도트풍 rect 기반 드로잉으로 자연스럽게 전환 가능
- GSAP의 `steps()` easing으로 레트로 애니메이션 즉시 적용 가능

### 도입 비용

| 작업 | 난이도 | 예상 범위 |
|---|---|---|
| Application 초기화 변경 | 낮음 | `Application.ts` 1파일 |
| 팔레트/폰트 교체 | 낮음 | `constants.ts` + CSS + 폰트 파일 |
| UI 컴포넌트 도트풍 전환 | 중간 | `Button.ts`, `ModeCard.ts`, `PickModeCard.ts` 등 6~7파일 |
| 이펙트 도트풍 전환 | 중간 | `effects/` 5파일 |
| 엔티티 도트 캐릭터 | 높음 | `Horse.ts`, `Marble.ts` + 스프라이트 에셋 제작 |
| 씬 배경/레이아웃 | 중간 | 각 Scene 파일 |

### 아키텍처 정합성

- 현재 `BaseScene → init() → update() → destroy()` 패턴은 변경 불필요
- `constants.ts`의 COLORS/FONT 상수를 교체하면 전체에 자동 반영
- 엔티티 교체 시 `Horse.container` 인터페이스만 유지하면 게임 로직 변경 없음

## 4. 권장안

### 1순위: 점진적 마이그레이션 (Phase 방식)

코드 기반(constants + 렌더링 설정 + Graphics)을 먼저 도트풍으로 전환하고, 이후 스프라이트 에셋을 도입하는 **2단계 전략**.

**Phase A — 코드 기반 도트 전환 (에셋 없이 즉시 가능)**

```
A-1. Application.ts 렌더링 설정 변경 (antialias, scaleMode, roundPixels, CSS)
A-2. constants.ts 팔레트/폰트 교체 (PICO-8 + Galmuri11)
A-3. UI 컴포넌트 도트풍 전환 (Button, ModeCard, PickModeCard, SectionLabel 등)
A-4. 이펙트 도트풍 전환 (Confetti, Countdown, ShakeEffect 등)
A-5. 엔티티 Graphics rect 기반 도트 캐릭터 (임시)
A-6. 씬 전환 효과 추가 (체커보드 와이프)
```

**Phase B — 스프라이트 에셋 도입 (Aseprite 제작 후)**

```
B-1. Aseprite로 캐릭터 스프라이트시트 제작 (말, 구슬 등)
B-2. AnimatedSprite 교체 + cacheAsTexture 적용
B-3. UI 패널용 NineSliceSprite 에셋 제작/적용
B-4. 배경/트랙 타일맵 도입 (선택)
```

### 2순위 (대안): 한번에 전체 교체

모든 에셋을 미리 제작한 뒤 일괄 교체. 완성도는 높으나 초기 투입 시간이 크고 중간 결과 확인이 어려움.

### 주의사항

- **스프라이트시트 패킹 시 1~2px 패딩** 필수 — nearest 모드에서 UV 정밀도 문제로 텍스처 경계 아티팩트 발생
- **CSS `pixelated` 정수 배율** — 소수점 배율에서 픽셀 크기 불균일 주의
- **한글 BitmapFont 지양** — 11,172자 텍스처 크기 폭발, 일반 Text + 웹폰트 사용
- **PixiJS v8 Custom Filter** — uniform 처리 변경으로 구현 난이도 높음, Graphics 기반 전환 효과 권장
