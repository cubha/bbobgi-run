## 1. Executive Summary

현재 구슬레이스는 **290px × 2400px 세로 단방향 트랙**으로, 실제 Marble Run의 핵심 재미요소(합류/분기/스파이럴/깔때기)를 살리지 못하고 있다. 리서치 결과, 다음 방향으로 전면 리디자인을 권장한다:

| 항목 | 현재 | 권장 변경 |
|---|---|---|
| 맵 크기 | 290px × 2400px | **1200px × 4000px** (X+Y 자유 스크롤) |
| 카메라 | Y축 단일 스크롤 | **CameraController 2축** (리더 추적 + 그룹 추적) |
| 트랙 구조 | 지그재그 램프 5-Zone | **모듈러 세그먼트** (깔때기/스파이럴/분기/합류/핀존) |
| 최소 구슬 | 2개 | **6개** (합류/분기 재미 최소 기준) |
| 구슬 크기 | 8px | **10~12px** (넓은 맵 대비 가시성 확보) |
| 렌더링 최적화 | 없음 | **Culling** (뷰포트 밖 세그먼트 비렌더링) |

## 2. 시장 분석 핵심 인사이트

### 2.1 Marble Run Satisfying 콘텐츠 트렌드

YouTube/TikTok "Marble Run Satisfying" 인기 콘텐츠에서 공통적으로 발견되는 요소:

| 요소 | 조회수 기여도 | 게임 적용 가능성 |
|---|---|---|
| **구슬 합류 장면** (여러 구슬이 한 곳으로 모임) | 매우 높음 | 깔때기(Funnel) 세그먼트 |
| **분기 후 재합류** (갈라졌다 다시 만남) | 높음 | Splitter + Merge 세그먼트 |
| **스파이럴/나선** (빙글빙글 회전) | 높음 | Spiral 세그먼트 |
| **계단식 캐스케이드** (통통 튀며 내려옴) | 중간 | Staircase 세그먼트 (현재 구현 있음) |
| **병목 통과** (좁은 곳 비집고 지나감) | 높음 | Bottleneck 세그먼트 (현재 구현 있음) |
| **도미노/연쇄 반응** | 중간 | 향후 확장 가능 |

**핵심 발견**: "구슬들이 **모이는** 장면"이 가장 높은 만족감을 유발한다. 이는 "예측 가능한 카오스"를 만들어 긴장감을 극대화하기 때문이다.

### 2.2 경쟁 제품 대비 차별화 기회

| 경쟁 제품 | 약점 | 뽑기런 차별화 |
|---|---|---|
| Marble Roulette (lazygyu) | 게임성 없음, 결과 연출 빈약 | 4가지 게임 모드, 시상식/벌칙 이분법 연출 |
| Marbles on Stream | PC+Twitch 한정, 오프라인 모임 부적합 | 웹 기반, 오프라인 모임/술자리 최적화 |
| Simple Marble Race | 관람형+조작 혼합 | 순수 관람형, 물리 기반 공정성 |

### 2.3 최적 구슬 수 시장 검증

| 구슬 수 | 합류/분기 재미 | 관전 가독성 | 적합 용도 |
|---|---|---|---|
| 2~3개 | 낮음 | 최고 | 결승 드라마용 |
| **4~5개** | 보통 | 높음 | 뽑기런 최소 허용 |
| **6~8개** | **높음** | 좋음 | **합류 재미 최적** |
| 10~12개 | 매우 높음 | 보통 | 대규모 이벤트 |

**결론**: 합류/분기 재미를 위한 최소 구슬 수는 **6개**. 2~5명 참가 시 더미 구슬을 추가하여 최소 6개 유지 권장.

### 3.2 카메라 시스템

**자체 CameraController 구현 권장** (pixi-viewport 대비 가벼움, 관람형에 최적)

```typescript
// camera/CameraController.ts
export class CameraController {
  private targetX = 0;
  private targetY = 0;
  private currentX = 0;
  private currentY = 0;
  private mode: 'leader' | 'group' | 'free' = 'group';

  constructor(
    private worldContainer: Container,
    private screenW: number,
    private screenH: number,
    private worldW: number,
    private worldH: number,
  ) {}

  /** 상위 1~3등 평균 위치 추적 */
  followGroup(marbles: Marble[]): void {
    const top = marbles.slice(0, Math.min(3, marbles.length));
    this.targetX = top.reduce((s, m) => s + m.body.position.x, 0) / top.length;
    this.targetY = top.reduce((s, m) => s + m.body.position.y, 0) / top.length;
  }

  /** 1등 단독 추적 (슬로모 구간) */
  followLeader(leader: Marble): void {
    this.targetX = leader.body.position.x;
    this.targetY = leader.body.position.y;
  }

  update(lerp = 0.08): void {
    this.currentX += (this.targetX - this.currentX) * lerp;
    this.currentY += (this.targetY - this.currentY) * lerp;

    // 월드 경계 클램프
    const halfW = this.screenW / 2;
    const halfH = this.screenH / 2;
    const cx = Math.max(halfW, Math.min(this.worldW - halfW, this.currentX));
    const cy = Math.max(halfH, Math.min(this.worldH - halfH, this.currentY));

    this.worldContainer.x = halfW - cx;
    this.worldContainer.y = halfH - cy;
  }
}
```

**컨테이너 구조 변경**:
```
container (씬 루트)
├── worldContainer (카메라 이동 적용)
│   ├── trackContainer (트랙 구조물)
│   └── marbleContainer (구슬)
└── hudContainer (화면 고정 UI)
    ├── timerBar
    ├── rankLabels
    └── miniMap
```

### 3.3 트랙 세그먼트 모듈화

현재 `buildZoneA()`, `buildZoneC()` 등 씬 내 직접 구현을 **세그먼트 클래스**로 분리:

```
src/
├── maps/
│   ├── TrackData.ts              # 맵 레이아웃 정의 (세그먼트 배열)
│   ├── TrackBuilder.ts           # 세그먼트 배열 → 물리+렌더링 빌드
│   └── segments/
│       ├── BaseSegment.ts        # 공통 인터페이스
│       ├── FunnelSegment.ts      # 깔때기 (합류/수렴)
│       ├── SpiralSegment.ts      # 스파이럴 (나선)
│       ├── SplitterSegment.ts    # 분기점 (Y자)
│       ├── PinZoneSegment.ts     # 핀존 (파친코)
│       ├── BottleneckSegment.ts  # 병목 통로
│       ├── RampSegment.ts        # 경사 램프
│       └── StaircaseSegment.ts   # 계단식 드롭
└── camera/
    └── CameraController.ts
```

### 3.4 핵심 구조물 Matter.js 구현 방식

| 구조물 | 구현 방식 | 물리 안정성 |
|---|---|---|
| **깔때기(Funnel)** | 두 기울어진 `Bodies.rectangle` 조합 (V자) | 높음 |
| **스파이럴** | 짧은 직선 세그먼트 16~32개를 나선 경로에 배치 | 중간~높음 |
| **분기(Splitter)** | 중앙 삼각형 정점 + 좌우 경사로 | 높음 |
| **합류(Merge)** | 깔때기 역형 (양쪽에서 중앙으로 수렴) | 높음 |
| **핀존** | 격자 `Bodies.circle(isStatic)` (현재 구현 있음) | 높음 |
| **루프** | 안쪽 곡면 세그먼트 8~16개 배치 | 낮음 (속도 요구 높음) — 비권장 |

### 3.5 렌더링 최적화: Culling

```typescript
// 매 프레임 카메라 업데이트 후
private cullSegments(): void {
  const margin = 100;
  const viewL = -this.worldContainer.x - margin;
  const viewR = viewL + this.screenW + margin * 2;
  const viewT = -this.worldContainer.y - margin;
  const viewB = viewT + this.screenH + margin * 2;

  for (const seg of this.segments) {
    seg.graphics.visible =
      seg.bounds.right > viewL && seg.bounds.left < viewR &&
      seg.bounds.bottom > viewT && seg.bounds.top < viewB;
  }
}
```

### 3.6 Matter.js 퍼포먼스 (대형 맵)

| 항목 | 현재 | 대형 맵 예상 | 대응 |
|---|---|---|---|
| 동적 바디(구슬) | ~10개 | 6~10개 | 변동 없음, 안전 |
| 정적 바디(트랙) | ~80개 | ~200개 | 정적 바디는 broadphase 비용 낮음, 안전 |

| 구슬-구슬 충돌 | 활성 | 활성 유지 | 10개 이하 안전 |

## 4. 더미 구슬 시스템

참가자가 6명 미만일 때 **더미 구슬**을 추가하여 합류/분기 재미를 보장한다.

| 참가자 수 | 더미 구슬 수 | 총 구슬 수 |
|---|---|---|
| 2명 | 4개 | 6개 |
| 3명 | 3개 | 6개 |
| 4명 | 2개 | 6개 |
| 5명 | 1개 | 6개 |
| 6명+ | 0개 | 참가자 수 그대로 |

더미 구슬 특징:
- 회색 계열 색상 (참가자 구슬과 시각적 구분)
- 이름: "●", "○" 등 심볼 표시
- 순위 판정에서 제외 (finishOrder에 포함하지 않음)
- 합류/병목 장면에서 물리적 존재감만 제공

## 5. 구현 우선순위

### Phase A: 아키텍처 리팩토링 (선행 필수)

1. **worldContainer / hudContainer 분리** — UI가 카메라와 함께 이동하는 문제 해결
2. **CameraController 추출** — X+Y 2축 카메라 독립 클래스
3. **Culling 시스템** — 세그먼트별 bounds 기반 visible 제어

### Phase B: 세그먼트 모듈화

4. **TrackSegment 인터페이스 정의** + BaseSegment 구현
5. **기존 Zone 코드 → 세그먼트 클래스 마이그레이션** (RampSegment, PinZoneSegment, BottleneckSegment, StaircaseSegment)
6. **TrackData.ts** — 맵 레이아웃 데이터 분리

### Phase C: 새 구조물 + 대형 맵

7. **FunnelSegment** (깔때기/합류)
8. **SplitterSegment** (분기점)
9. **SpiralSegment** (나선)
10. **대형 맵 레이아웃** 구성 (1200px × 4000px, 분기/합류 포함)

### Phase D: 더미 구슬 + 폴리싱

11. **더미 구슬 시스템** (6개 최소 보장)
12. **미니맵** (정적 트랙 텍스처 + 동적 구슬 마커)
13. **구슬 크기 조정** (8px → 10~12px)
